设计模式的总结

策略模式

###https://www.cnblogs.com/java-my-life/archive/2012/05/10/2491891.html
一个具体的类，类中定义 一个具体去做动作的类和方法（该方法调用具体去做动作的类去实现动作片）

Strategy是一个抽象类或者接口，里面有抽象方法需要具体类去实现

Context根据传入不同的Strategy实现类，使得contextInterface的实现方式不同。

#简单工厂模式

传入具体的 参数对象去构建具体对象（参数可以灵活设定）但是会出现A参数CPU和B参数主板的组合。

#抽象工厂模式

传入具体的构造工厂去构建具体对象（内部参数已经在具体的构造工厂里设定好）A工厂只会造出A参数CPU和A参数工厂。

Spring

<context:component-scan base-package=”com.xx.xx” />
context:annotation-config/
<aop:aspectj-autoproxy /> 开启@aspectj的使用，Spring默认不支持这个，需要使用这个声明去开启。

context:component-scan/不但启用了对类包进行扫描以实施注释驱动 Bean 定义的功能，同时还启用了注释驱动自动注入的功能（即还隐式地在内部注册了 AutowiredAnnotationBeanPostProcessor 和 CommonAnnotationBeanPostProcessor），因此当使用 context:component-scan/ 后，就可以将 context:annotation-config/ 移除了。

而context:annotation-config/ 主要功能是注册了
AutowiredAnnotationBeanPostProcessor,
CommonAnnotationBeanPostProcessor,
PersistenceAnnotationBeanPostProcessor, RequiredAnnotationBeanPostProcessor

所以开启了注解扫描基本上可以忽略context:annotation-config/

#Redis的两种持久化类型
AOF
可以持久化每行命令，并且可以设定 按秒持久化，按操作次数持久化。
优势：不易丢失数据，即使宕机也只会丢失两次持久化之间的数据。
劣势：效率低，空间占用过大，当数据量过大时候影响redis性能

RDB（快照）
另一个线程不停地把redis存储的信息写入磁盘。
优势：
恢复和迁移容易，
避免大量的IO（直接全部写入了，相比于AOF不用不停地读写）

劣势：
如果定时备份的时候宕机，上次备份到这次备份之间的数据会全部丢失
由于是通过子进程来做同步的，如果数据量较大时，同步可能会带来服务器暂停几秒。

#Redis集群
参考链接
1 https://www.cnblogs.com/linjiqin/p/7451822.html
2 https://www.jianshu.com/p/14835303b07e
最少三主三从 如果一台主节点挂了，而没有从节点被选举为主节点，则整个集群不用。

集群有几种分片方式
1.客户端分片 简单，性能高 但是多业务使用的时候可维护性很差。
2.基于代理的分片 维护成本增加，并且增加了代理中间件，多了一层转发，多了一层消耗
3.路由查询
将请求发送到任意节点，接收到请求的节点会将查询请求发送到正确的节点上执行。
例如： Redis-cluster

####Redis-cluster集群介绍
客户端路由.
Redis 集群中内置了 16384个哈希槽，当需要在 Redis 集群中放置一个 key-value 时，redis 先对 key 使用 crc16 算法算出一个结果，然后把结果对 16384 求余数，这样每个 key 都会对应一个编号在 0-16383 之间的哈希槽，redis 会根据节点数量大致均等的将哈希槽映射到不同的节点。

####Redis集群投票机制
Redis集群中有多台redis服务器不可避免会有服务器挂掉（主从模式的时候）。redis集群服务器之间通过互相的ping-pong判断是否节点可以连接上。如果有一半以上的节点去ping一个节点的时候没有回应，集群就认为这个节点宕机了。

#基于Redis做的分布式锁
基于SETNX命令 去SET 一个 key value
如果存在则返回false 如果不存在则设置成功 返回true
一般 key 为锁id value 为锁的过期时间 同时也需要去设定一个过期时间.
（当忘记系统中断，忘记设定过期时间，可能会出现死锁，所以我们每次都会不断对比value里的过期时间，所以其实每个key value分别是有两个过期时间的，一个是基于redis的，另外一个是基于value值里的）
去获取锁的时候，考虑到上锁者可能因为某种原因忘记设定redis的过期时间。所以会去比对设置在value上的过期时间，如果当前时间大于value上的过期时间，就去getset获取锁（getset可以同时返回目前最新的value值）
如果getset返回的那个value值和你上一次获得的value值相同，则说明你成功获取了锁。
否则说明在你获取锁的同时，有另外一个人也在争抢锁并且先你一步获取锁
(疑问解答：getset命令只要执行了其实就已经成功的刷新了value，但是由于你和另一个争抢者set的时间差不多，所以不会有什么影响)

#Redis 常见问题

####1.缓存穿透
查询一个 一定不存在的key时候 会造成缓存穿透，每次都会去查数据库
解决方法：

  1. 布隆过滤器。 对所有可能查询的参数以hash形式存储，在控制层先进行校验，不符合则丢弃。还有最常见的则是采用布隆过滤器，将所有可能存在的数据哈希到一个足够大的bitmap中，一个一定不存在的数据会被这个bitmap拦截掉，从而避免了对底层存储系统的查询压力。
  2. 将这个空key存入redis中，设定一个短的过期时间。

####2.缓存雪崩
一定时间内大量的缓存同时过期，导致大量的读库及更新缓存操作。

解决方法：

1.将数据设定不同的过期时间，避免同时大量过期

2.可以通过缓存reload机制，预先去更新缓存，再即将发生大并发访问前手动触发加载缓存

3.在缓存失效后，通过加锁或者队列来控制读数据库写缓存的线程数量。比如对某个key只允许一个线程查询数据和写缓存，其他线程等待。

4.做二级缓存，一级缓存失效后去读二级缓存。一级缓存的过期时间短，二级缓存的过期时间长。
